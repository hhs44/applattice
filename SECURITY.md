# 安全策略

## 支持范围

安全修复面向最新发布版本和 `main` 分支。尚未发布稳定版之前，`0.x` 版本可能发生不兼容变化；旧的开发快照不承诺单独维护。

## 报告漏洞

请不要通过公开 Issue、Pull Request、讨论区或聊天公开漏洞细节。

优先在 GitHub 仓库的 **Security → Advisories → New draft security advisory** 中提交私密报告。若仓库尚未启用私密漏洞报告，请仅创建一个不含技术细节的 Issue，请求维护者建立私密联系渠道。

报告最好包括：

- 受影响的版本、组件和部署方式；
- 可重复的最小验证步骤；
- 影响范围和可能的利用条件；
- 已知缓解措施；
- 报告者希望使用的署名方式。

维护者的目标是在 3 个工作日内确认收到报告，在完成影响评估后协商修复和披露时间。该时间是协作目标而非服务等级承诺。

## 重点安全边界

- Gateway 是浏览器访问业务后端的唯一入口，未匹配权限规则的请求必须拒绝。
- `AUTH_MODE=dev` 仅用于本地开发，生产环境必须使用受信任的 OIDC 配置。
- 远程前端清单、离线包、npm/Python 制品和容器镜像在生产交付前应进行摘要或签名校验。
- 不要在应用目录、日志、OpenAPI 示例或错误响应中放置凭据和敏感业务数据。

公开发布后，仓库管理员还应启用 GitHub Secret Scanning、Push Protection、Dependabot alerts 和 Code Scanning。
