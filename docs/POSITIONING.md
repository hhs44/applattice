# AppLattice 项目定位

## 一句话定义

AppLattice 是一个开源的企业级可组合应用平台开发与交付套件：它让多个自治的全栈应用共享统一入口、身份、体验和治理规则，同时保留独立仓库、技术栈和发布节奏。

中文主张：**让应用保持自治，让平台保持统一。**

英文描述：**An open-source composable enterprise application platform kit for unified experience, autonomous delivery, contract governance, and air-gapped environments.**

## 它面对的真实矛盾

企业系统演进通常会在两种极端之间摇摆：

- 所有页面和接口都进入一个大仓库，早期简单，后期耦合、发布互相阻塞，AI 也需要读取过大的上下文。
- 每个团队完全独立建设，交付灵活，但登录、导航、权限、主题、接口接入和运维体验不断重复。

AppLattice 不试图消灭这种张力，而是给它一条清晰边界：平台集中管理跨应用公共能力，业务团队自治管理业务能力，双方通过可版本化协议组合。

## 核心价值

### 对平台团队

- 将入口、身份、权限、导航、主题和接入策略沉淀为一次建设、持续复用的平台能力。
- 用应用目录、服务目录和契约锁替代手工代理配置与口头约定。
- 用标准脚手架建立安全默认值和最低质量基线。
- 让新增应用不再要求修改门户业务源码。

### 对业务团队

- 保留独立仓库、后端语言、数据库、测试和发布流水线。
- 用独立开发壳获得正常的 Vite HMR，不依赖门户才能开发页面。
- 通过 SDK 和清单接入平台，不需要理解门户内部实现。
- 单个应用可以独立升级、回滚或暂时下线。

### 对架构与治理

- 把隐式依赖转化为 Manifest、OpenAPI、协议版本和制品摘要。
- 通过 Gateway 建立统一安全边界，对未声明访问默认拒绝。
- 支持从模块化单体、存量系统到自治应用的渐进迁移。
- 将平台控制面与业务数据、规则和 SLO 的所有权分开。

### 对 AI 辅助开发

- 一个业务需求通常只需读取目标应用、Manifest、OpenAPI 和 SDK 接口。
- 一个平台需求通常只需读取 Portal、Gateway、目录和相关契约。
- 跨仓制品避免让 AI 追踪整个 Monorepo 的内部源码关系。
- `AGENTS.md` 和目录边界把最小上下文变成工程约束，而不是使用习惯。

### 对内网和受限环境

- 平台包可以固化为带摘要的本地 tgz。
- JavaScript 与 Python 依赖可以提前准备为 pnpm Store 和 wheels。
- 容器镜像可以从本地或内网 registry 获取，不要求公共镜像仓库。
- 离线模式显式禁止公网访问并报告缺失制品。

## 目标用户

1. 正在维护多个后台系统，希望建立统一门户的平台工程团队。
2. 需要同时支持 Python、Node 和存量后端的企业研发团队。
3. 对内网、离线或受监管交付有要求的组织。
4. 正从巨型前端或单仓系统迁向混合仓库和自治交付的团队。
5. 希望降低 AI 编程上下文成本、明确代码所有权边界的团队。

## 适用场景

- 企业内部管理平台和业务工作台。
- 质量工程、测试、运维和研发效能平台。
- 数据门户、报告中心和多业务域运营后台。
- 多个既有系统的统一入口与渐进式整合。
- 受限网络中的多团队应用交付基线。

## 非目标

AppLattice 首版不提供：

- 开箱即用的具体业务产品；
- 拖拽式低代码业务建模；
- Kubernetes 控制面或完整 PaaS；
- 完整 IAM、服务网格、日志平台或制品仓库；
- 替代每个业务服务自身的领域设计、数据库治理和发布责任。

## 与相邻概念的区别

| 概念           | AppLattice 的关系                                                |
| -------------- | ---------------------------------------------------------------- |
| 微前端框架     | 使用其运行时组合能力，但额外提供 Gateway、治理、脚手架和离线交付 |
| 内部开发者平台 | 提供应用接入与交付基线，但首版不是基础设施自助服务门户           |
| API Gateway    | 包含面向浏览器的 BFF/Gateway，但产品价值不止流量代理             |
| Monorepo 模板  | 平台本身是 Monorepo，业务应用采用独立仓库的混合模式              |
| 测试平台       | 可由 AppLattice 构建的一类业务平台，不再是项目本身的定义         |

## 产品原则

1. **组合优先于合并**：运行时组合应用，不把业务源码重新集中进门户。
2. **契约优先于约定**：可机器验证的清单和 Schema 优先于文档约定。
3. **自治不等于无治理**：业务独立交付，但遵守入口、安全和协议基线。
4. **本地优先，部署可选**：先保证开发体验，再生成 Docker 等交付能力。
5. **离线是设计条件**：不把断网支持当作发布后的补丁。
6. **渐进迁移**：允许存量服务适配，不要求一次性重写。

## 建议成功指标

- 从空目录到首个可运行 Portal 的时间。
- 从空目录到首个已注册全栈 App 的时间。
- 新增应用所需的门户业务源码改动数，目标为零。
- 契约破坏在发布前被发现的比例。
- 单个远程应用故障时门户和其他应用的可用性。
- 干净断网环境中的生成、安装和构建成功率。
- AI 完成单应用需求时需要读取的仓库与文件数量。

## 名称解释

“Lattice”意为格构或晶格：节点保持独立，连接关系形成稳定结构。AppLattice 表达的不是一个巨型应用，而是一组自治应用通过平台协议构成统一系统。这与项目的技术边界和组织价值一致。
