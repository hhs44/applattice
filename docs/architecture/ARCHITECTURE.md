# 智能测试平台架构设计（重设计版）

## 1. 结论

平台采用“体验面、业务控制面、执行面、数据面、横切治理”五类架构分区，不再把所有能力画成一条自上而下的运行时链路。

关键设计决策：

1. 存量端到端业务继续稳定运行，通过防腐层适配器渐进接入，不以整体重写为前提。
2. 新增贯通业务流在完成实现、验证和接入门禁后，注册并发布到平台。
3. 确定性的长流程业务编排与非确定性的 AI Agent 编排使用不同引擎和治理方式。
4. 控制指令、业务事件和大体量传感器数据使用不同通道，避免传感器数据穿越普通 API 网关。
5. 领域服务初期采用边界清晰的模块化单体；只有在团队、扩缩容或发布节奏出现独立需求后再拆分服务。

## 2. 范围与假设

### 范围

- 统一门户、角色工作台和开放接口；
- 存量端到端业务维护与渐进接入；
- 新增贯通业务流的设计、验证、注册和发布；
- 需求、场景、用例、执行、评估、缺陷、报告等领域能力；
- LangGraph AI Agent、RAG、模型接入和人工审核；
- HIL / SIL / MIL、仿真器、实车和物理台架执行；
- ADAS 多传感器数据接入、处理、资产化和存储；
- 可观测性、安全合规、数据治理与工程交付。

### 当前假设

- 平台优先面向企业内网用户，数据包含车辆、传感器和测试结果等敏感信息。
- 存量业务不能一次性停机迁移，需要并行运行和渐进适配。
- 业务流程可能持续数分钟至数天，需要状态持久化、重试、补偿和人工确认。
- 传感器原始数据体量远大于业务元数据，需要独立的数据通道和生命周期策略。
- 可用性、吞吐量、部署规模和多租户边界尚未量化，文档中不虚构具体 SLO 数值。

## 3. 系统上下文

### 主要用户

- 测试负责人：关注业务流运行、质量风险、资源和交付结果。
- 测试工程师：设计用例与场景、执行测试、分析结果并闭环缺陷。
- 业务流负责人：维护存量端到端业务，设计和发布新增贯通业务流。
- 平台管理员与安全审计人员：管理权限、发布、合规证据和运行治理。

### 外部依赖

- 存量端到端业务系统；
- 缺陷、项目、售后等企业系统；
- HIL / SIL / MIL、仿真器、实车和物理台架；
- Camera / LiDAR / Radar / GPS-IMU / CAN / ETH 数据源；
- LLM、Embedding、Reranker 等模型服务；
- SSO、消息通知和企业基础设施。

## 4. 架构分区与职责

| 分区 | 核心组件 | 主要职责 |
|---|---|---|
| 体验面 | 统一门户、角色工作台、AI Copilot、开放 API | 提供统一入口和角色化任务视图，不承载核心业务规则 |
| 接入与治理 | API Gateway / BFF、IAM、业务流注册与发布门禁、契约与 Schema Registry | 认证授权、聚合接口、流量治理、版本与接入质量控制 |
| 业务控制面 | 业务流程编排引擎、领域能力中心、AI Agent 编排、集成适配中心 | 管理确定性业务流程、领域规则、AI 任务和存量系统适配 |
| 执行面 | 执行调度、资源管理、执行 Worker、台架/仿真/实车连接器 | 将测试计划转换为可追踪的执行任务，处理超时、重试和资源隔离 |
| 数据面 | 事件总线、流式接入、时间同步与解析、数据工厂、资产目录、湖仓与多模存储 | 承载业务事件与大体量测试数据，提供契约、质量、血缘和生命周期治理 |
| 横切治理 | 可观测性、安全合规、数据治理、CI/CD 与平台工程 | 覆盖全部运行组件，不作为业务请求必经的串行层 |

## 5. 核心容器

### 5.1 业务流程编排引擎

负责可重复、可审计的长流程业务状态：流程版本、定时器、重试、补偿、人工任务、发布和回滚。候选实现可以是 Temporal、Camunda 或 Flowable，最终选型应通过 ADR 和验证性原型确定。

业务流程编排不得依赖 LLM 的非确定性输出维持核心状态一致性。

### 5.2 AI Agent 编排

LangGraph 负责需求分析、场景生成、结果辅助分析、报告生成和人工审核等 AI 任务。模型输出必须经过结构化校验；高风险动作只生成建议或待确认命令，由业务流程引擎和权限系统决定是否执行。

### 5.3 领域能力中心

按业务边界组织模块：

- 需求与策略；
- 用例与场景；
- 执行计划与资源；
- 结果与评估；
- 缺陷与闭环；
- 报告与项目。

初期在一个可部署单元中保持独立模块、独立数据访问接口和明确所有者，避免共享数据库成为无边界的数据池。

### 5.4 集成适配中心

采用 Ports and Adapters / 防腐层模式，为存量业务和外部系统提供：

- API 适配；
- 事件适配；
- 数据映射；
- 身份与权限映射；
- 幂等、重试、限流和熔断；
- 接入健康度与迁移状态。

### 5.5 执行调度与 Worker

调度层只管理命令、状态和资源，不传输大体量传感器数据。每个执行任务必须具有全局 `run_id`、幂等键、超时、取消和重试策略。Worker 通过连接器控制台架、仿真器或实车，并把结果元数据与数据位置写回平台。

### 5.6 数据平台

- 业务事件：通过事件总线发布，采用至少一次投递时必须保证消费者幂等。
- 传感器数据：通过流式接入、分片上传或对象存储签名地址进入数据面。
- 数据契约：定义 Schema、语义、事件时间、摄取时间、新鲜度、完整性和所有者。
- 数据质量：校验时间同步、丢包、帧率、Schema 漂移、空值、范围和业务不变量。
- 数据资产：统一场景、用例、信号字典、知识图谱、Corner Case、报告和证据链的目录与版本。

## 6. 关键业务流

### 6.1 存量端到端业务维护与接入

1. 存量系统继续拥有原业务流程和数据。
2. 适配器暴露稳定 API 或事件契约，并记录映射和幂等策略。
3. 平台先获得统一可见性，再逐步复用领域能力。
4. 仅当能力所有权和迁移收益明确时，才将局部能力迁入平台。

### 6.2 新增贯通业务流发布

1. 业务流负责人设计流程并绑定领域能力。
2. 平台校验流程完整性、API/事件契约、权限、监控和回滚方案。
3. 在隔离环境运行端到端验证和回归测试。
4. 通过门禁后注册版本并灰度发布。
5. 运行中产生统一业务追踪 ID、事件和审计记录。

### 6.3 测试执行

1. 业务流程引擎创建执行计划。
2. 执行调度器根据资源、环境和优先级分配任务。
3. Worker 控制台架、仿真器或实车。
4. 状态与结果元数据通过 API/事件回传；原始数据直接进入数据面。
5. 结果评估、缺陷闭环和报告由领域能力和 AI 辅助共同完成。

### 6.4 AI 辅助

1. 业务流程提交结构化 AI 任务。
2. LangGraph 调用 RAG、模型网关和工具。
3. 输出经过 Schema 校验、敏感信息检查和证据引用。
4. 需要执行副作用的动作进入人工确认或业务流程审批。

## 7. 接口与契约

| 类型 | 使用场景 | 必须定义 |
|---|---|---|
| REST / gRPC | 查询、短事务命令、资源管理 | 版本、授权、幂等键、错误码、超时、分页 |
| 领域事件 | 状态变化、跨域通知、多消费者订阅 | Schema、事件 ID、聚合 ID、发生时间、顺序、重放策略 |
| 工作队列 | 执行任务、耗时处理 | 任务 ID、租约、重试、死信、幂等和取消 |
| 数据契约 | 测试数据集、传感器数据、特征和标签 | Schema、语义、时间口径、质量阈值、所有者、变更策略 |

所有关键调用和事件使用 `correlation_id`；测试执行使用全局 `run_id`；业务流版本使用不可变 `flow_version`。

## 8. 数据所有权

| 数据 | 所有者 | 主存储 |
|---|---|---|
| 业务流定义与版本 | 业务流管理模块 | PostgreSQL |
| 需求、用例、场景元数据 | 对应领域模块 | PostgreSQL |
| 执行计划与运行状态 | 执行领域模块 | PostgreSQL / 状态存储 |
| 高频执行指标 | 执行与可观测模块 | 时序数据库 |
| 原始传感器与大型结果文件 | 数据平台 | 对象存储 |
| 向量与检索索引 | AI / 知识服务 | 向量数据库与搜索索引 |
| 审计与合规证据 | 安全合规模块 | 不可篡改或受控审计存储 |

禁止多个模块直接修改同一业务表；跨域更新通过公开接口或事件完成。

## 9. 可靠性与可运维性

- 所有写操作支持幂等或明确不可重试。
- 长流程使用持久化状态、超时、重试、补偿和人工干预。
- 事件消费者支持重复消息、乱序和重放。
- 数据管道支持 Checkpoint、回放、补数、坏数据隔离和血缘追踪。
- 关键路径具备日志、指标和 Trace，统一关联 `correlation_id` / `run_id`。
- 发布流程为 build → test → package → deploy → verify → rollback。
- 高风险能力使用功能开关、灰度发布和可验证回滚。

## 10. 主要风险与缓解

| 风险 | 影响 | 缓解措施 |
|---|---|---|
| 用 LangGraph 承载确定性业务状态 | 状态难以审计和补偿 | 业务流程引擎与 AI Agent 编排分离 |
| 原始传感器数据走普通 API | 网关拥塞、成本和稳定性问题 | 元数据走 API，原始数据走数据面 |
| 存量系统直接共享平台数据库 | 边界失控、迁移困难 | 防腐层适配、契约化接入、数据所有权 |
| 过早拆分微服务 | 运维成本和分布式故障增加 | 模块化单体优先，按证据拆分 |
| 异步事件无幂等和重放策略 | 重复执行、状态不一致 | 事件 ID、Inbox/Outbox、幂等消费者 |
| AI 输出直接执行高风险动作 | 错误调度或数据破坏 | Schema 校验、权限检查、人工审批 |

## 11. 演进路线

### 阶段 1：建立平台骨架

- 统一门户、IAM、API Gateway / BFF；
- 业务流注册与发布门禁；
- 领域能力模块边界与统一追踪 ID；
- 存量业务适配器和统一可观测性。

### 阶段 2：打通业务闭环

- 引入确定性业务流程编排；
- 打通需求、场景、执行、评估、缺陷和报告；
- 建立事件契约、幂等和补偿机制；
- 新增业务流通过验证后接入。

### 阶段 3：建设数据面与 AI 能力

- 建立传感器数据直达通道、Schema Registry 和数据质量；
- 建设数据资产目录、场景库和 Corner Case 库；
- LangGraph、RAG、模型网关和 AI 评测治理接入业务流程。

### 阶段 4：规模化与合规

- 根据瓶颈和团队边界拆分服务；
- 完善 SLO、容量、灾备、灰度和成本治理；
- 建立安全威胁模型、合规证据链和审计闭环。

## 12. 待确认问题

- 单日测试运行数、传感器数据量和峰值带宽；
- 目标可用性、恢复时间和恢复点目标；
- 是否存在多租户、跨组织或外网访问；
- 哪些存量端到端业务必须保持原数据所有权；
- 业务流程引擎的团队技术栈和许可证限制；
- 功能安全和数据合规需要覆盖的具体系统边界。

## 13. 工程仓库与交付边界

规模化阶段采用 ADR-0005 定义的混合仓库：统一门户、Gateway/BFF、UI/SDK、服务目录、契约锁和环境编排位于平台 Monorepo；达到独立团队、发布、扩缩容、故障隔离、技术栈或安全边界门槛的业务服务位于独立仓库。

服务仓库拥有业务实现、数据迁移和 OpenAPI/Event Schema；平台通过版本与 SHA-256 锁定契约，通过镜像 digest 组装环境。开发者可用本地路径组合多个检出目录，CI 和生产不得依赖其他仓库的未发布源码。完整容器、接口、风险和开发路径见 `hybrid-repository.mmd`、`hybrid-development-path.mmd` 与 ADR-0005。
