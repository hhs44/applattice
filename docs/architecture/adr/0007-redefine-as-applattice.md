# ADR-0007：将项目重新定义为 AppLattice

- 状态：Accepted
- 日期：2026-07-22

## 背景

项目最初从智能化测试平台场景出发，逐步实现了统一门户、Gateway、动态远程前端、Python/Node 全栈模板、应用与服务目录、契约锁、多仓联调和离线制品。当前能力已经不依赖测试领域模型，真实解决的问题是多个自治业务应用如何共享统一体验与治理，同时保持独立交付。

继续使用“智能化测试平台”定位会造成三个问题：潜在使用者误以为这是测试管理产品；通用架构能力被具体业务场景遮蔽；包名、镜像和脚手架默认值持续固化错误边界。

## 决策

项目命名为 **AppLattice**，定位为“企业级可组合应用平台开发与交付套件”，核心主张是“让应用保持自治，让平台保持统一”。

采用以下命名规则：

- GitHub 仓库使用 `applattice`；
- npm 包使用 `@applattice/*`；
- 镜像示例使用 `applattice/*`；
- 脚手架默认门户使用通用的 `enterprise-platform` 和“企业应用平台”；
- Schema 示例域使用 `applattice.example`；
- 历史测试领域材料保留在 `docs/architecture/archive`，不再作为当前规范。

当前产品边界包含 Portal、Gateway/BFF、运行时微前端组合、应用/服务目录、契约治理、双层脚手架、本地多仓联调和离线交付基线。测试、运维、数据等是可承载场景，不属于平台内置业务。

## 后果

正面影响：

- 项目名称、架构和实际能力保持一致；
- 面向平台工程、内部应用整合和内网交付的用户群更清晰；
- 业务应用不再被测试领域术语限制；
- 包、制品和文档形成可持续的统一品牌命名空间。

代价与兼容性：

- `@platform/*` 到 `@applattice/*` 是包名变更，需要重新安装依赖和发布制品；
- 已生成应用需要更新依赖、vendor tgz 和导入路径；
- 旧仓库 URL 依赖 GitHub 重定向，使用者应更新 remote；
- 历史文档会保留旧名称，需要清楚标识其归档状态。

## 迁移

1. 重命名仓库与公开元数据。
2. 更新门户、CLI、包作用域、镜像、Schema 和示例配置。
3. 重建锁文件与 Todo 验证应用的离线 tgz。
4. 更新定位、架构、贡献和发布文档。
5. 运行契约、类型、测试、构建、格式和示例应用门禁。

已有外部使用者升级时，应将所有 `@platform/*` 依赖切换为对应的 `@applattice/*` 版本，并重新生成或替换 vendor 制品。

## 备选方案

- 保留测试平台名称：与通用能力不一致，拒绝。
- 使用泛化的 `Enterprise Platform Kit`：描述准确但缺少可识别品牌，拒绝。
- 按单一技术特征命名为 Microfrontend Portal：忽略 Gateway、契约、脚手架和离线交付价值，拒绝。
