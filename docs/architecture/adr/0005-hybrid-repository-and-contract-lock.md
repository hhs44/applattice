# ADR-0005：平台 Monorepo 与独立服务仓库并存

- 状态：采纳
- 日期：2026-07-20
- 替代：ADR-0004 中“前端与全部后端长期位于同一 Monorepo”的部分

## 背景

平台已经具有统一门户、Gateway/BFF、共享 UI/SDK 和模块化后端样例。随着团队、Python/Node 技术栈、发布节奏和扩缩容需求分化，把全部服务继续放在一个仓库会放大流水线、权限、依赖升级和 AI 上下文成本；立即拆成完全自治的多仓又会失去统一联调、契约和内网交付能力。

## 决策

采用混合仓库：

- `platform-core` Monorepo 拥有 Portal、Gateway/BFF、UI、SDK、服务目录、契约锁、集成测试和环境编排；
- 每个达到拆分门槛的业务服务拥有独立仓库、数据、OpenAPI/Event Schema、流水线和不可变镜像；
- API 契约由生产者服务仓库拥有并发布，平台通过 `contracts.lock.json` 固定名称、版本和 SHA-256，不复制可变的内部实现包；
- 本地工作区使用不提交的 `workspace.local.json` 指向多个检出目录，生成 Compose 叠加文件；CI 与生产只消费已发布契约和镜像；
- 浏览器仍只访问 Gateway；Gateway 通过服务 ID 和版本化目录解析上游，不向门户暴露拓扑；
- `apps/domain-service` 仅作为迁移兼容样例，新服务默认由独立 Node/Python 仓库模板创建。

## 接口与所有权

| 边界        | 生产者          | 消费者                | 治理方式                            |
| ----------- | --------------- | --------------------- | ----------------------------------- |
| OpenAPI     | 业务服务仓库    | BFF、生成 SDK         | SemVer、摘要锁、兼容性检查          |
| 事件 Schema | 事件生产者      | 多个服务              | Schema Registry、向后兼容、重放测试 |
| OCI 镜像    | 各应用/服务仓库 | 环境发布仓库          | commit SHA/digest、签名、SBOM       |
| 服务目录    | 平台仓库        | BFF、联调和发布工具   | 代码审查、责任人、健康检查、超时    |
| 数据        | 单一业务服务    | 只经公开 API/事件访问 | 禁止跨库写、迁移向后兼容            |

## 影响

正向影响：服务可以按团队和语言独立演进；平台 CI 不再构建所有业务服务；AI 任务可限制在单仓；本地与内网仍可一键组合；发布清单可以精确回滚单个镜像。

代价：需要维护服务目录、契约兼容门禁和跨仓版本升级；本地开发需要多个检出目录或预构建镜像；跨仓变更不能依赖一个原子提交，必须使用扩展/迁移/收缩和消费者测试。

## 风险与缓解

| 风险                 | 缓解                                                             |
| -------------------- | ---------------------------------------------------------------- |
| 契约文件复制后漂移   | 摘要锁、生产者导出、CI 校验、版本制品                            |
| 多仓联调不可复现     | 本地工作区文件不入库，服务目录和镜像版本入库，Compose 由工具生成 |
| BFF 变成业务逻辑中心 | 仅允许认证、聚合和适配；业务规则及数据留在领域服务               |
| 跨仓发布顺序导致中断 | Expand → Migrate → Contract，兼容窗口、功能开关、消费者测试      |
| 内网缺少公网站点     | npm/PyPI/OCI/OS/模型/漏洞库全部镜像化，运行时禁止公网            |
| 多仓增加 AI 搜索范围 | 根与仓内 `AGENTS.md`、任务最小上下文、契约而非源码作为跨仓上下文 |

## 被否决方案

- 所有代码永久保留在一个仓库：团队和发布边界成熟后，变更与权限耦合继续扩大。
- 每个前端模块也立即独立仓库：当前统一体验和发布收益高于微前端自治收益。
- 单独建立手工维护的“全局契约仓库”：会弱化生产者所有权并产生双写；采用制品注册表加平台契约锁。
