# ADR-0004：统一门户、BFF 与 Monorepo 模板

- 状态：已被 ADR-0005 部分替代；统一门户与 BFF 决策仍有效
- 日期：2026-07-20

## 背景

现有业务集中在一个前端，但后端能力由多个系统提供。团队需要在内网快速开发和部署，又不能让浏览器依赖后端服务拓扑，也不宜在业务边界尚未稳定时承担全面微前端和微服务治理成本。

## 决策

- 以 React 门户作为统一体验壳，承担菜单、会话和平台级交互；
- 以 Fastify Gateway / BFF 作为浏览器唯一 API 入口，承担 OIDC、RBAC、聚合和适配；
- 前端与初期后端使用 pnpm Monorepo，共享版本化契约、SDK 和 UI 组件；
- 领域服务采用模块化单体，按独立团队、发布、扩缩容和故障隔离证据拆分；
- 交付同时支持内网制品库、Docker Compose 和离线镜像包；
- 生产环境禁止开发身份模式。

## 影响

### 正向

- 浏览器不感知微服务地址和接口差异；
- 单条命令可启动完整开发环境，新模块可一次生成三层接入点；
- API Schema 在前后端共享并执行运行时校验；
- 保留模块独立部署和微前端演进空间。

### 代价

- Gateway 成为关键运行组件，需要高可用和容量治理；
- Monorepo 需要统一依赖升级和变更门禁；
- 拆分服务时需要把进程内模块接口转换为版本化网络契约。

## 被否决方案

- 浏览器直接调用所有后端：暴露服务拓扑，认证、错误和跨域治理分散；
- 立即全面微前端化：当前缺少独立团队和发布节奏证据，运行时复杂度收益不足；
- 继续扩展单一巨石前端：模块所有权和独立演进能力会持续下降。
