# 智能测试平台业务层架构重构说明

## 设计目标

业务层不再假设所有流程已经统一，也不要求存量端到端业务一次性迁移。平台通过“稳态维护、能力沉淀、渐进接入”三条路径持续演进：

- 存量端到端业务保持独立闭环和稳定运行，在维护过程中逐步标准化接口并复用平台能力。
- 平台沉淀业务流编排与集成中枢，统一处理流程、事件、状态和规则。
- 新增贯通业务流先在平台外完成实现与验证，达到接入标准后再注册、发布到平台。
- 需求、用例、场景、执行、评估、缺陷、报告和项目等能力以领域服务形式复用，避免每条业务流重复建设。

## L3 业务层结构

### 1. 存量端到端业务

定位为“在运业务区”，重点是持续维护、稳定性和兼容性。接入平台时优先增加适配器、API 或事件契约，不以整体重写为前提。

### 2. 业务流编排与集成中枢

作为业务层核心，负责：

- 流程定义、编排和版本管理；
- API 调用与事件路由；
- 长流程状态、超时、重试和补偿；
- 业务规则、权限和发布控制；
- 统一业务追踪 ID 与端到端运行视图。

### 3. 新增贯通业务流

采用“实现 → 验证 → 注册 → 发布”的接入路径。流程在具备稳定的端到端行为后进入平台，避免将未成熟逻辑直接耦合到平台主干。

### 4. 可复用领域能力

将需求/策略、用例/场景、执行调度、结果评估、缺陷闭环、报告/项目等能力服务化，由存量业务和新增业务流共同调用。

## 建议的业务流接入门槛

一条业务流进入平台前，至少应满足：

1. 明确流程负责人、参与角色、服务等级和数据责任。
2. 主流程、异常流程、超时重试与补偿路径均已验证。
3. API、事件和数据模型已版本化，并具备兼容策略。
4. 权限、审计、敏感数据处理和安全边界已确认。
5. 日志、指标、链路追踪和业务告警已经接入。
6. 具备自动化回归、灰度发布和回滚方案。

## 演进原则

- 存量业务“可维护优先”，不以迁移数量作为平台化成果。
- 新业务流“贯通后接入”，不把平台当作流程试验场。
- 共性能力进入领域能力中心，个性流程留在业务流编排中。
- 通过契约和事件解耦流程与底层系统，避免重新形成大单体。
