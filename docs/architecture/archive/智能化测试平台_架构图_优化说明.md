# 智能测试平台架构图优化说明

## 架构判断

原图已经覆盖门户、网关、业务闭环、AI 编排、ADAS 数据管道、测试资产、数据存储、工程效能与基础设施，主干完整；可观测性、安全合规和数据治理也以横切能力呈现，方向合理。

主要问题集中在三类：

1. “平台 A / 平台 B / 双平台融合”把组织或历史系统边界带进目标架构，削弱了统一平台的领域视角。
2. L3 能力说明越出容器，MCP 工具列表溢出到右侧安全栏，底部存在较大无效留白。
3. CI/CD 被连续箭头表现为运行时数据链路，语义不够准确；“功能安全合规达标”也容易被理解为平台本身即可完成认证。

## 本次已落地优化

- 将整体叙事收敛为单一“智能测试平台”，删除双平台、平台 A、平台 B、双后端等信息。
- API 网关统一为“核心业务路由、测试服务路由、AI 服务路由”。
- L3 以分析、设计、执行、评估、报告和缺陷闭环表达测试域能力，不再按来源平台拆分。
- 修正 L3 与 L4 重叠、MCP 文本横向溢出、底部留白和弱文字对比度问题。
- 将 CI/CD 明确为工程支撑，并用独立的“工程交付”连线区别于运行时层间调用。
- 将“功能安全合规达标”改为“合规证据链完备”，更准确地表达平台对 ISO 26262 / SOTIF 的支撑边界。
- 增加 SVG 标题、描述与中文字体回退，便于无障碍访问和跨环境渲染。

## 后续架构建议

- 补充接口与数据契约治理：OpenAPI、AsyncAPI、Schema Registry、版本兼容策略。
- 明确控制面与数据面：业务请求经 API 网关；传感器大数据宜通过消息总线或对象存储直达数据管道，避免穿越普通业务 API。
- 为 AI 能力补充评测与治理：Prompt / 模型版本、离线评测集、效果门禁、成本与安全策略。
- 将需求、用例、场景、执行、结果、缺陷和安全证据统一到可追溯 ID，形成真正的闭环证据链。
